从UI设计角度谈谈H5设计

2018-04-06

主要通过：

  • H5是什么？
  • 我们为什么需要 H5 ？
  • 从UI设计角度来说，我们需要做什么？

这三个方面来聊聊 H5 设计。

H5是什么？

H5定义：”超文本标记语言或超文本链接标示语言，是目前网络上应用最为广泛的语言，也是构成网页文档的主要语言“（网上查的）

我们为什么需要H5？

总结了一下，H5具有以下几个优点：

效率高

iOS 和安卓发版都需要一定的时间，举个例子，如果运营小伙伴需要上活动，就需要移动端在活动约定时间之前发版，而且需要再提前一定的时间，向各大应用商店提交发版需求，平台审核通过以后，才能完成最终的活动上线需求。

相比较之下，如果是通过 H5 上架活动的话，直接就可以发布了，而且也可以快速的迭代。

跨平台

是的，只要有浏览器就可以用，而且也可以嵌入到移动端里面，从 app 打开，和原生开发的没有太大差别。

成本低

因为有了以上两个优点，自然开发成本就低了，不论是人员成本还是时间成本。

从UI设计角度来说，我们需要做什么？

UI设计不仅仅是涉及到美观问题，还会在一定程度上影响用户体验性。那么，针对 H5 的优点以及特性，主要从以下几个方面来谈谈 UI 需要注意的问题：

  • 设计尺寸
  • 设计规范
  • 交互

设计尺寸

以“750x1334”为例：

实际可设计的区域就是去掉 title 栏剩下的区域。也就是说，因为需要考虑跨平台等第三方接入的问题，所以我们不能自己定义 title 栏，比如微信里面最常见的“更多”，已经放在了右上角。

其他关于组件的设计尺寸，比如列表、按钮等，如果没有特殊要求（或者说不知道要定义多大的话）则可以参考 ios 里面的 UI 组件规范。不过，最好还是能够装 “ ps play ” 进行实时查看，以便得到较优的视觉体验。

字体也是同样的道理，这里就不再啰嗦了。

设计规范

我们都知道，设计规范不仅对设计方面的迭代有好处，还能对开发方面的版本迭代有好处。

某些组件可以参考 ios/md 里面的规范，但是大部分的设计规范最好还是自己定义，因为 ios/md 里面的设计规范，主要是针对原生 app 开发的。但是，此时我们做的是 H5 页面设计，还涉及到“跨平台”的问题。

  • 比如，关于字体的问题，原生 app 里面是“你想多少都行”，但是 H5 有最小的字体限制，最好不要小于 20px ——来自前端小哥的温馨提示。

  • 再比如，关于切图标注的问题，此处推荐蓝湖，当然，和蓝湖一样的神器还有 zeplin ，但是由于 zeplin 服务器不在我大天朝，因此设计稿同步以及数据读取等协作使用起来没有蓝湖顺畅（亲测）。

温馨提示：
无论是蓝湖还是 zeplin 都是跟本地的 ps 画板联动的，也就是说，如果设计稿换了画板/ ps 重装了，那么两个都会在同步时都会受到影响，最好分模块建画板。

交互

也是由于“跨平台”这一原因，在交互方面，不需要太突出的“平台”特性，除非是嵌入到原生 app 当中。

  • 例子1

    在原生 app 当中，以 ios 为例，左滑动删除操作是很常见的，但是如果被嵌入浏览器/微信等第三方中，滑动触发的可能是页面切换操作，因此，要慎重考虑是否选用原生的 “左滑删除”操作。

  • 例子2

    日期选择器，原生 app 都可以调用各自的原生组件，但是如果嵌入到浏览器等第三方中，应该选用较通用的设计，这一个问题可以先咨询一下前端小哥，问问有哪些常见的样式。

  • 例子3

    关于列表方面的，层级最好不要太深。原生app是有一个保存到本地的，但如果被嵌入第三方，会涉及到信息储存的问题，比如进行填表单的步骤，如果层级太深，一不小心就关闭了页面，那么先前被填入的信息就被清空了，这样的用户体验想想就知道多抓狂了。

写在最后的话

无论是围绕什么平台进行设计，都离不开“体验良好”这一核心，因为一切（开发、设计、产品等）都是以帮助使用者又快又好完成相关进程为目的的。

多与产品链上其他的小伙伴进行交通，可以获得多角度的结论。再运用到自己的设计中，才能进一步推动产品的产出以及与小伙伴的协作，亲测有效。

比如，一个标签，到底是后台推送，还是前端判断？如果是后台推送，那么需要考虑标签内容长短、使用情况再进行较为通用的设计；一组数据包含几个类型的值，后台是使用了一个字段来表示这一组数据，还是单独使用了不同的字段存储对应类型的值？如果是只用了一个字段来表示这一组值，那么，如果我们 UI 完全不知情,然后为了让信息更加清晰地显示而把这些值分到不同模块中，无疑会增加了前端的开发量，因为除了判断数据，还得把它们分开放到每个模块中……

所以，还是得多了解产品链上的其他工作进程，比如可以通过多体验自家产品的管理后台，去看看其中的数据是怎么存放、定义的，这样，再回到 UI 层就会考虑更多的东西了。当然，后台设计不友好就要另外考虑了，不能因为适应后台而忘了设计对产品的重要意义。
